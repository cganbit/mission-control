Arquitetura de 3 camadas — custo de contexto
Camada 1
Pré-provisionados .claude/agents/ · 4 agentes
ZERO custo

Claude Code invoca automaticamente ao detectar a task. Rodam em subprocess isolado — não consomem contexto do agente principal. Contexto do projeto embutido: VPS, tabelas, auth patterns, gotchas históricos.

Camada 2
On-demand Agent tool · 4 agentes | .skills/ · 16 skills
ZERO custo

Agentes (security-auditor, explorer-agent, database-architect, performance-optimizer) invocados manualmente via Agent tool com template em .skills/AGENT_INVOCATION.md. Skills carregadas sob demanda — saem do contexto naturalmente.

Camada 3
Catálogo SKILLS_CATALOG.md · 1x por sessão
~100 linhas

Arquivo leve lido no início de sprints. Índice completo de todas as skills, agentes e knowledge bases com instrução de quando carregar cada um. Skills do projeto (n8n-gotchas, apple-parts, etc.) carregadas apenas quando relevante.

Por que isso é boa prática de mercado
✓ Single Responsibility

Cada agente tem um domínio claro. Nenhum tenta fazer tudo. O debugger conhece os gotchas do projeto; o devops conhece os containers; o frontend conhece os padrões de UI.

✓ Context Injection

Contexto do projeto embutido no agente, não passado manualmente toda vez. O agente já chega "onboarded" — conhece o VPS, as tabelas, os padrões de auth.

✓ Auto-routing

Claude Code detecta a tarefa e invoca o especialista automaticamente via o campo description do frontmatter. Sem instrução manual em cada task.

✓ Tiered Specialization

Pré-provisionados (tarefas frequentes) vs on-demand (raras). Economia de contexto e recursos. Padrão documentado no Multi-Agent Systems Guide da Anthropic.
