🚀 Deploy Evolution — Antes vs Depois

Migração de paramiko + secrets.py para rsync + SSH key · Sessão 16 · 24/03/2026

Fluxo comparado
Antes — paramiko + secrets.py
pip install paramiko → secrets.py manual → tar.gz inteiro → SFTP upload → docker run inline → credenciais no CMD
Depois — rsync + SSH key + docker compose
bash deploy_mc.sh → rsync delta → verificar hash → docker compose build → docker compose up → ✅ confirmação hash
⚠️
Antes
deploy_mc.py + paramiko
📦
Dependência externa: paramiko
Precisa instalar manualmente. Quebra em máquina nova, CI/CD, ou após limpeza de venv.
pip install paramiko
🔑
secrets.py gitignored
Arquivo com senha do VPS, DB, JWT, API keys. Precisa ser copiado manualmente para cada PC. Sem ele, deploy falha silenciosamente.
import secrets as S # quebra com módulo built-in!
🗜️
Upload de tarball completo
Toda sessão: compacta TUDO (exceto node_modules), faz upload via SFTP. Lento mesmo para mudança de 1 linha.
🐳
docker run com credenciais inline
Senha do banco, JWT, API keys visíveis no comando e no histórico de processos do VPS.
docker run -e JWT_SECRET="abc123" -e PG_PASS="..." ...
❌
Sem verificação de hash
Não há confirmação de que a imagem foi realmente atualizada. Deploy "bem sucedido" pode rodar código antigo.
Riscos: senha em texto claro no CMD • deploy falha entre máquinas • código antigo em produção sem aviso • secrets.py nunca sincronizado
✅
Depois
deploy_mc.sh + rsync + SSH key
🔐
SSH key — zero senha no código
Chave SSH já configurada no VPS. Nenhum arquivo de credentials necessário. Funciona em qualquer máquina com a chave.
ssh root@187.77.43.141 # sem senha
⚡
rsync delta — só o que mudou
Rsync compara checksums e envia apenas arquivos alterados. Deploy de 1 arquivo: segundos. Deploy completo: mais rápido que tarball.
rsync -az --delete --exclude=node_modules ...
🐳
docker compose + .env no VPS
Credenciais ficam no .env persistido no VPS. docker compose lê automaticamente. Nunca passadas na linha de comando.
docker compose up -d # lê .env local
🔍
Verificação de image hash
Compara hash antes e depois do build. Se igual → alerta imediato. Impossível rodar código antigo sem saber.
Hash antes: sha256:a1b2c3... → depois: sha256:d4e5f6... ✅
🧹
Sem dependências externas
Bash nativo + rsync (disponível em qualquer Linux/Mac/WSL). Sem pip, sem bibliotecas, sem setup.
bash scripts/deploy_mc.sh # só isso
Ganhos: zero senha em código • funciona em qualquer máquina • deploy incremental • confirmação de hash • credenciais seguras no VPS
Comparação técnica
Aspecto Antes (paramiko) Depois (rsync)
Autenticação VPS Senha em secrets.py Chave SSH (zero senha)
Credenciais app Passadas inline no docker run .env no VPS, lido pelo compose
Velocidade de sync Tarball completo sempre Delta — só arquivos alterados
Dependências paramiko (pip install) rsync + bash (nativo)
secrets.py Obrigatório, gitignored, esquecido Não necessário
Verificação de build Nenhuma Hash antes vs depois
Portabilidade Só Windows com paramiko Qualquer OS com bash/WSL
Histórico de processos Senhas visíveis no ps aux Nenhuma credencial exposta
Como migrar — checklist
  • Confirmar que docker-compose.yml existe em /opt/mission-control/ no VPS
  • Confirmar que .env existe em /opt/mission-control/.env no VPS com todas as variáveis
  • Testar: ssh root@187.77.43.141 "echo ok" — deve funcionar sem senha
  • Rodar: bash scripts/deploy_mc.sh — deve completar sem pedir senha
  • Verificar hash diferente na saída do script
  • Verificar curl https://mc.wingx.app.br/api/health responde 200
  • Arquivar deploy_mc.py e secrets.py (manter como fallback por 1 sprint)
